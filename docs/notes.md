# Deep Code Review Notes\n\n## Repository trajectory\n\nThe codebase has evolved from a streamable HTTP server concept into a CLI-first autonomous agent architecture. Documentation previously lagged this shift and referenced old run paths.\n\n## Current architecture (validated)\n\n- Entrypoint: `cmd/fire_starter/main.go`\n- Agent loop and model/provider integration: `src/agent/workflow.go`\n- Decision matrix and tool contract generation: `src/matrix/tool_registry.go`\n- Technique dispatch and module execution: `src/matrix/real_executor.go`\n- Engagement memory and extraction: `src/matrix/knowledge_graph.go`\n\n## Strengths observed\n\n1. **Solid package-level test coverage**\n   - Most matrix/agent/module paths have direct tests.\n   - Edge-case tests are present for payload normalization, phase gating, and intelligence extraction.\n\n2. **Clear execution decomposition**\n   - `tool_registry` handles naming/schema exposure.\n   - `real_executor` handles routing and module invocation.\n   - `knowledge_graph` centralizes state and extraction.\n\n3. **Deterministic phase controls**\n   - Advancement and submission require explicit evidence and phase coverage.\n   - Tool scoring includes anti-repeat pressure via completed-tool penalty.\n\n4. **Resilience in extraction layer**\n   - Knowledge graph handles valid JSON and malformed text outputs.\n   - Concurrent update behavior is tested.\n\n## Key gaps / risks\n\n1. **Environment mismatch risk**\n   - `go.mod` requires `go 1.26.2`; older local toolchains fail quickly.\n\n2. **Documentation drift (fixed in this pass)**\n   - Legacy references to `./src/http-server` were outdated.\n\n3. **Large switch-based dispatcher maintenance cost**\n   - `src/matrix/real_executor.go` has many technique-specific branches; future growth will increase complexity and merge friction.\n\n4. **Technique naming strictness**\n   - Dispatch relies on exact normalized technique strings; matrix changes can silently route to default behavior if not aligned.\n\n## Recommendations (next iteration)\n\n- Introduce a registration map for technique -> module factory to reduce switch complexity.\n- Add a matrix/dispatcher consistency test that asserts every decision technique maps to a known execution branch.\n- Add a short CI pipeline with `go test ./...` and Go version pinning to prevent toolchain drift.\n\n## Docs refresh completed\n\n- Updated `README.md` to reflect real architecture, build/run flags, and review findings.\n- Rewrote `docs/quick_start.md` for CLI-first execution and troubleshooting.\n- Rewrote `docs/test.md` into an actionable verification guide aligned with current tests and entrypoint.\n\n## Validation run\n\n- `go test ./...` passes in the current environment used for this review.\n
