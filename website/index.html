Platform Features

Built for local, inspectable assessments

Fire Starter keeps orchestration, evidence, and outputs inside your environment so you can validate what happened and why.

Self-hosted runtime

Build and run the agent locally with Go. Use hosted model providers or point `-provider local` at an OpenAI-compatible endpoint while keeping scan execution in your own environment.

Modular Go techniques

Add a technique entry in `src/matrix/decisions.json`, implement the module in `src/modules/core`, and register it with `RegisterModule(...)` to make it available to the runtime.

Traceable evidence

The workflow keeps a live knowledge graph in memory, stores findings in `fire_starter.db`, and writes `fire_starter_report.md` when the engagement concludes.

Autonomous Workflow

A phase-aware assessment loop

The agent scores available tools per target, updates the knowledge graph after execution, and advances through the engagement based on evidence.

1. Reconnaissance

Bootstraps target knowledge through discovery-oriented techniques such as search, subdomain work, and initial surface mapping.

2. Scanning

Expands context with route checks, port discovery, and error analysis to identify reachable attack surfaces.

3. Vulnerability Analysis

Validates likely weaknesses, enriches context, and promotes promising leads into deeper investigation.

4. Exploitation

Runs active technique modules, records proof-of-concept evidence, and updates database-backed findings.

5. Reporting

Generates a markdown report and preserves the final knowledge graph state for review.

Built-in Coverage

Broad default technique set

The repository ships with a large set of built-in modules across reconnaissance, analysis, exploitation, and post-exploitation workflows.

Authorization and logic

  • BOLA, BFLA, and IDOR validation paths
  • Mass assignment and business-logic bypass checks
  • Rate-limit and access-boundary probing

Web and API input abuse

  • SQL, NoSQL, XPath, LDAP, and command injection modules
  • GraphQL introspection and advanced GraphQL checks
  • HTTP verb tampering, smuggling, and parameter pollution

Session and browser security

  • JWT, OAuth, SAML, CSRF, and session-fixation testing
  • CORS, security headers, and cookie handling analysis
  • XSS and DOM-based XSS coverage

Discovery and infrastructure

  • Google dorking, subdomain enumeration, and takeover analysis
  • Port scanning and route discovery helpers
  • Local privilege escalation and SSH pivot modules

Under the Hood

Designed around registries and shared state

The CLI entrypoint, orchestration workflow, module registry, executor, knowledge graph, and TUI are cleanly separated so contributors can change one layer without rewriting the rest.

Architecture overview

Flags and config are loaded in `cmd/fire_starter/main.go`. The agent workflow in `src/agent/workflow.go` initializes the provider, scores tools, and manages target progression. The executor in `src/matrix/real_executor.go` resolves techniques to registered module factories under `src/modules/core`, while `src/matrix/knowledge_graph.go` and `src/matrix/db.go` retain evidence.

Adding a new built-in check

Current runtime support is centered on the built-in registry in `src/modules/core`.

1
Define the technique Add metadata in src/matrix/decisions.json.
2
Implement the module Create the Go implementation in src/modules/core/.
3
Register and test Register it with RegisterModule and add `_test.go` coverage.
Factory registration pattern
func init() {
    RegisterModule("my_module", func(payload map[string]any, onLog func(string)) (ExecutableModule, error) {
        target := PayloadString(payload, "url", "http://127.0.0.1")
        module := &MyModule{Target: target}
        return ModuleWrapper{
            Module: module,
            ExecuteFunc: func(ctx context.Context) (any, error) {
                return module.Execute(ctx)
            },
        }, nil
    })
}
